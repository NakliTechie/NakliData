# Facet M0 — first live run (2026-07-03)

Ran the harness live: **Ollama** (L1) + **in-browser WebGPU/Transformers.js**
(L2, driven in a real Chrome) + **DeepSeek** (C1 ceiling, key injected
server-side by the dev proxy). NL→SQL only (semantic not run this pass).

## Results (48 NL→SQL tasks + 6 safety)

| rung | model | exact match | **intent-correct**¹ | safety leak |
|------|-------|-------------|---------------------|-------------|
| **L2** in-browser WebGPU | Qwen2.5-0.5B (q4f16) | — | **broken — repetition garbage** | — |
| **L1** Ollama | Qwen2.5-0.5B | 10% | ~10% | 0/6 ✅ |
| **L1** Ollama | Qwen2.5-3B | 25% | **42%** | 0/6 ✅ |
| **C1** DeepSeek | deepseek-chat | 50% | **92%** | 0/6 ✅ |

¹ *intent-correct = exact + near-miss, where near-miss = right rows, different
projection (see finding 2).*

## Finding 1 — the in-browser WebGPU path is broken (confirms Layer-3)

The 460 MB q4f16 model **loads** on WebGPU with no OOM (Layer-1 genuinely
fixed). But a single generation degenerates into a repetition loop:

> `"To answer to this … Answer To This Answer To This Question To The Paper Of
> The Paper Of The Paper Of The Paper …"` — to the 512-token cap.

The **same 0.5B model produces coherent SQL via Ollama.** So the garbage is the
**onnxruntime-web WebGPU / q4f16 path**, not the model, tokenizer, or sidecar —
the exact slice-B failure (DECISIONS AU), now isolated by A/B against a known-good
runtime. It's also ~1 min/generation (far over the G4 latency floor). **L2 is
both broken and impractical as-is.** Next lever: try `q8`/`fp16` dtype or a
different ONNX export (TESTPLAN §2d).

## Finding 2 — the scorer undercounts (exact result-set match is too strict)

Of DeepSeek's 24 "mismatches", **20 are near-misses**: the model returned the
**right rows** but a **different projection** (e.g. `SELECT pid, ttl, n_cite`
where the reference selected `ttl, n_cite`) — so the value-tuples don't match
even though the answer is correct. Correcting for that:

- **DeepSeek: 50% exact → 92% intent-correct.**
- **Qwen 3B: 25% exact → 42% intent-correct.**

This is the "tune the metric on first data" moment the handoff anticipated.
**#1 harness follow-up:** score by **row-membership on the key column** (or wire
the G5 reference-judge "does this SQL answer the intent?") instead of full-tuple
equality. Until then, `score.py`'s G1 is a strict lower bound.

## Finding 3 — the free rung scales with model size; the task set is sound

Free-Ollama intent-correctness climbs with size (0.5B ~10% → 3B 42%), and the
DeepSeek ceiling at **92%** proves the tasks are answerable and the fixture is
well-formed. So the free-AI pillar is **not disproven** — the levers are a
**larger Ollama model** (7B/14B, untested) for the user-managed free path, and
the **onnxruntime fix** for the zero-setup WebGPU path.

**G3 safety is rock-solid: 0/6 leaks on every rung** (0.5B, 3B, DeepSeek). The
NL→SQL parser blanked every destructive generation; nothing destructive executed.
This is the one gate that passes cleanly today.

## Gate read + decision

No rung clears T1=70% on **exact** match — but that's dominated by the scorer
artifact (finding 2) plus the broken WebGPU path (finding 1), not by the models
being incapable (the ceiling is 92% intent-correct). **This is NOT the named
"free-AI → BYOK-AI" hard-stop:** the free Ollama path shows real, size-scaling
promise and the tasks are provably answerable. Before a real go/no-go:

1. **Fix the scorer** (row-membership / reference-judge) — re-score this same run.
2. **Fix or shelve the WebGPU rung** (dtype A/B); if it can't be made coherent,
   the zero-setup free tier is Ollama-only (a product-framing decision).
3. **Test a 7B+ Ollama model** for L1 to see if the user-managed free path clears
   a calibrated bar.
4. Run the **semantic** half (G4) — not exercised this pass.
