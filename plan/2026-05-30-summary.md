## Decisions (non-obvious)

- **Drop pre-fetch SRI on cross-origin DuckDB-wasm load.** The blob-pre-wrap it required broke cross-blob worker access in current Chrome. New trust boundary: version-pin in URL + build-time SHA-384 verify against `integrity.json` when bytes are vendored. Spec amendment **A14**.
- **Vendored DuckDB is the default, CDN is opt-in (`?cdn=1`).** Reverses the original assumption. The vendored bytes ship with every Pages deploy; CDN path now used only for users explicitly opting in.
- **GH Pages serves as canonical mirror for non-vendored deploys.** Cloudflare's 25 MiB per-file limit can't host the 34 MB WASM; `.assetsignore` skips `duckdb-fallback/` and the runtime cross-fetches from `naklitechie.github.io/NakliData/duckdb-fallback/`. Bytes were vendored + SHA-verified at build time → trust is "we built and uploaded these," not third-party CDN trust.
- **Excel via SheetJS, not DuckDB excel extension.** The extension isn't published for our DuckDB-wasm revision (v1.1.1/wasm_eh). Earlier "no SheetJS" rule was about CDN block; npm-vendored SheetJS lazy chunk has no such issue. ~330 KB lazy chunk, only loads when user mounts xlsx.
- **Cohort cell is a separate `kind`, not `kind: 'sql'` with a flag.** Cleaner discriminated union; persistence-shape clarity; UI variant (COHORT label + user-count badge) flips cleanly.
- **Funnel + Path are custom SVG, not Observable Plot.** Matches existing `pie` pattern; ~5 KB each in the main bundle vs a chunk fetch; lets us own the visual identity (funnel drop-off annotations specifically).
- **Three-axis framing for the Databricks comparison.** personal↔team × workbench↔platform × thin-client↔thick-platform. NakliData is top-left; Databricks/Snowflake/Fabric are bottom-right. The category-vs-category clarity drives "we borrow ergonomics, not architecture" decisively.

## Tried then rolled back

- **W1.8.x deploy debugging arc — the load hang.** Started by assuming SRI-fetch + blob-wrap of DuckDB worker bytes was working; turns out a Worker spawned from blob A cannot `fetch` sibling blob B from the parent's blob registry in current Chrome. Three failed fixes (drop chain, broaden CSP, defer revoke) before realising the fundamental constraint and switching defaults to the vendored path. Cost ~30 min of round-trip-to-deploy debugging.
- **Chrome MCP `file_upload` with `~/Downloads/...` path.** Tool restricts uploads to user-shared paths (Claude-UI attachments + designated session folders). Workarounds explored (base64 inline via javascript_exec — too large; copy to `~/.claude/.../uploads/` — still rejected; gist hosting — too much setup) all hit dead-ends. Parked: user attaches via UI OR clicks Add file manually.
