## Testing / review checklist before next development

Run from a fresh clone on the desktop. The order matters — `smoke` is
the one most likely to catch regressions that `tsc` and `vitest` miss.

1. `npm install` — postinstall vendors DuckDB-wasm into
   `public/duckdb-fallback/` and writes `integrity.json`. Watch for
   SHA-384 hash mismatches if the lockfile pulled a different version.
2. `npm run check` — `tsc --noEmit` + `biome check`. Should be clean,
   ≤ 14 biome warnings.
3. `npm run test` — vitest. 60 unit + 4 e2e expected green.
4. `npm run smoke` — headless Playwright. **Run this even if everything
   else is green.** It catches CSP / FSA / worker-bootstrap / classifier
   regressions that the other gates let through.
5. `npm run build && ls -la dist/` — confirm `dist/index.html` ≤ 600 KB
   (currently ~320 KB) and `dist/chunks/codemirror.js` exists as a
   separate lazy chunk (~370 KB).
6. **Manual schema-panel pass** (CLAUDE.md stop-checklist item 5):
   render a workbook, click a column header, override a column's
   classification, confirm the override sticks across a re-render.

## Sandbox gotcha to be aware of

This sandbox blocks `extensions.duckdb.org`, so any Theme 1 mount that
needs a DuckDB extension (SQLite, Excel, SPSS/SAS/Stata) fails locally
in this environment but works fine in the user's browser. **The desktop
session won't hit this** — but if smoke fails on a mount and the
network looks suspicious, that's the cause. Theme 1 wave 3 (vendoring
extensions) closes the gap.
