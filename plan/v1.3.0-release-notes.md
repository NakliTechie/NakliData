## TL;DR

The largest release to date. v1.3.0 lands **two full handoff tracks** plus
the UI that makes them user-visible, a **whole-codebase hardening audit**,
and the **fully-in-browser local model runtime**:

- **Lakehouse Parity (v1.2 M1–M5)** — anonymized export sink, cell
  lineage tracker, incremental refresh, sidecar auto-visualization,
  visual query builder.
- **Prior Art (v1.3 M0–M6)** — measures layer, associative cross-filter,
  stats cell, report layout + print-to-PDF, shelf-based chart authoring,
  lineage edit mode. Shipped as data-layer + tests first…
- **Phase 2 UI** — …then wired user-visible: cross-filter grey-out, an
  inter-cell **associations** panel, shelf drop-zones, and lineage
  edit mode.
- **Forward-pass audit** — 3 Critical + 16 High + ~30 supporting findings
  fixed across 6 themed commits; a CI **verify gate** now blocks deploys.
- **Local model runtime (W3.2 slice B)** — Transformers.js + OPFS-cached
  weights, a fourth BYOK provider that runs entirely in the tab.

Feature release. **No breaking changes** (all additive; the one config
change is the shell bundle budget raised 600 → 750 KB, amendment A30).

## Headline — Lakehouse Parity (v1.2 M1–M5)

### Anonymized Export Sink (M1)
Sixth sink on every result. Per-column `keep` / `hash` / `redact` /
`bucket` / `drop`, defaults driven by §3.2 sensitivity badges. Applied as
a DuckDB SQL projection rewrite (md5 + DATE_TRUNC + FLOOR), every
identifier + literal through a dedicated quoter — airtight against hostile
column names. Per-export salt via `crypto.getRandomValues`, shown once,
never persisted. JSON manifest written alongside (records strategies +
`saltUsed: boolean`, never the salt). Amendment A25.

### Cell Lineage Tracker (M2)
"Where does this number come from?" Every SQL/cohort/assertion cell
records upstream inputs by walking the DuckDB `EXPLAIN (FORMAT JSON)` plan
(not regex) — CTE-shadow-safe, captures inline `read_parquet()` paths.
Accessible list view + hand-rolled three-lane SVG (no D3). Persists into
`.naklidata`. Amendment A26. *(See the v1.3.0 lineage fix below.)*

### Incremental Refresh (M3)
"Refresh" header button runs one change-detection sweep: FSA folders
re-fingerprint (size + mtime, no reads); HTTP fires a HEAD (ETag /
Last-Modified / Content-Length). Modal shows changed sources + downstream
cells (cascaded via the M2 lineage graph) + uncheckable sources.
**No background polling.** Amendment A27.

### Sidecar Auto-Visualization (M4)
"Suggest chart" chip → the sidecar returns a strict JSON `ChartProposal`
(8 chart types + x/y/group + title), materialized as a chart cell.
**No prose**; hallucinated columns drop the whole proposal. Amendment A28.

### Visual Query Builder (M5)
Form → SQL via a strict no-string-concat emitter (type-validated
per-column quoters; hostile values drop the filter or land in one quoted
literal). Single table + optional single JOIN, no nested subqueries.
Output goes to a SQL cell — you click Run. Amendment A29.

## Headline — Prior Art (v1.3 M0–M6)

### M0 — extraction-readiness sweep
Engine-boundary lint: the pure-logic core (`src/core/*`) has no DOM/FSA/
browser globals. `taxonomy/load.ts` takes an injected fetcher;
`anonymize.ts` moved to core; `ChartConfig` extracted to its own module
(one schema, three producers).

### M2 — Measures Layer
Named, versioned semantic metrics referenced via `MEASURE(name)` in SQL.
Macro-expands to a filtered-aggregate SQL fragment
(`SUM(amount) FILTER (WHERE status='completed')`). Single audited
expansion point.

### M1 — Associative Cross-Filter
Qlik's associative model on DuckDB. Click a value → SELECTED / ASSOCIATED
/ EXCLUDED states; excluded values render **greyed, not hidden**
(absence-as-signal). Selection store persists to `.naklidata`; type-correct
SQL literals (H13). **Phase 2 UI shipped this release** (below).

### M4 — Stats cell
Descriptive statistics + a correlation matrix over an upstream result, all
in DuckDB SQL. No regression / modelling.

### M3 — Report layout + print-to-PDF
Paginated report cell (KPI tiles + cell embeds). Export = browser
print-to-PDF with scoped `@media print` CSS (no pdf-lib). Cell-ref content
embedded on `beforeprint`, restored on `afterprint`.

### M5 — Shelf-based chart authoring · M6 — Lineage edit mode
Pure logic + tests shipped here; **both wired user-visible in Phase 2**
(below).

## Headline — Phase 2 UI (this release made v1.3 user-visible)

- **Cross-filter grey-out (M1).** Clicking a result value paints
  associative states in place; non-co-occurring values grey out. Computed
  in JS over the materialised result (intra-cell) — no engine round-trip.
  Surgical repaint preserves scroll + focus.
- **Associations panel (M1).** New header button. **Hybrid** authoring:
  auto-suggested links (columns sharing a taxonomy type or name across
  cells) + a manual link form + active-links list. Linking two columns
  cross-filters across cells — a selection in one greys non-co-occurring
  values in the other. Inter-cell reuses the intra-cell engine by
  propagating selected values across linked columns (DECISIONS AE).
  Persists to `.naklidata`.
- **Shelf-based chart authoring (M5).** A Manual | Shelves toggle on the
  chart cell; drag fields onto Columns / Rows / Color drop-zones (DnD +
  keyboard select). Shelves COMPILE to the same `ChartConfig` the manual
  controls write (Transparency Rule). `inferFieldClass` reads the result
  data; identifier-on-y is rejected with a teach toast.
- **Lineage edit mode (M6).** Edit | Done toggle on the lineage panel;
  insert-on-edge + delete-node (dependents listed before confirm) wired to
  `applyCanvasOp`, persisted via the store. Reposition deferred
  (layout-only no-op).

## Headline — Hardening (forward-pass audit)

A fresh-eyes whole-app audit (`plan/forward-pass-2026-06-10.md`) surfaced
3 Critical · 16 High · 33 Medium · 24 Low · 18 Stray. Closed across 6
themed commits. Highlights:

- **C1** — the v1.1 "Explain this error" sidecar button was unreachable
  since it shipped (a vestigial `hidden`); un-hidden + Chrome-verified.
- **C2** — spec amendment A30 (the 600 → 750 KB bundle cap the gate
  already pointed at) was never written; now documented.
- **C3** — the M6 round-trip invariant test was tautological; rewritten to
  go through the real `lineageGraphFromJson` validator.
- **H1–H5** — FSA-handle IDB leak, `?lens=` embedded-SQL surfacing, a 2 MB
  gzip-bomb cap on shared links, a lineage-walk cycle guard, non-finite
  LIMIT rejection.
- **CI verify gate** — `deploy.yml` now runs build → check → test → smoke
  on every push + PR *before* build/deploy.

### Lineage fix — view-backed mounts (`de1a309`)
Found while verifying M6: a plain `SELECT * FROM invoices` recorded **no**
lineage. Root cause — every CSV/JSON/Parquet/Iceberg source is mounted as
a DuckDB VIEW, and duckdb-wasm 1.29.0 inlines views at bind time + emits
space-padded scan op names with no file path, so the physical-plan walk
returned `[]`. Fix: trim op names, accept `extra_info.Text`, and union a
catalog-filtered CTE-aware SQL sniff into the plan walk (preserving the
§M2 CTE-shadow guarantee). A smoke regression guard now asserts source→
cell lineage on a live run.

## Headline — Local model runtime (W3.2 slice B)

A fourth BYOK provider that runs **entirely in the browser** via
Transformers.js, with weights cached in OPFS. Curated model list
(Qwen2.5-1.5B default / Phi-3.5-mini / Llama-3.2-1B) with size + license +
quality hints, live download progress, per-model cache delete, and a
boot-path auto-load when weights are already cached. Lazy chunk (525 KB),
zero bytes paid unless the user picks Local. Amendment A24.

> **Owed:** manual per-job validation of all 6 sidecar jobs against a
> live local model (`plan/w32-slice-b-validation.md`) needs WebGPU and
> can't run headless. Shipped unvalidated by decision (DECISIONS AG) — the
> code is unit-tested and the provider is opt-in; a v1.3.1 closes the QA.

change. Pre-v1.3 `.naklidata` files round-trip cleanly; the new
`measures` / `selections` / `associations` / `lineage` fields are all
optional.
