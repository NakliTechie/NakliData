# NakliData v1.5.1 — release notes

Tag `v1.5.1` at `origin/main`. The second surface of the **Resolve track** —
**M2: the segment primitive** (the *Audience* verb of `resolve → segment → own`).
Additive minor; versioning is git-tag-based (`BUILD_VERSION` stays `0.1.0`).

## Headline — Segments (`SEGMENT(name)`)

A named, reusable **boolean predicate** over a table — a saved "audience" —
managed in the Semantic layer panel next to measures and dimensions:

    high_value_lapsed = total_amount > 100000 AND last_seen < '2026-01-01'

Reference it in a WHERE clause and it expands — at the SAME single audited point
as `MEASURE()` / `DIM()` — to the parenthesized predicate:

    SELECT * FROM invoices WHERE SEGMENT(high_value_lapsed)
    → SELECT * FROM invoices WHERE (total_amount > 100000 AND last_seen < '2026-01-01')

- **One expansion point.** `SEGMENT(name)` joins `MEASURE(name)` + `DIM(name)` in
  the same depth-capped `expandMeasures` pass — a segment can reference a
  dimension/measure and vice-versa. No new SQL dialect.
- **Honest failure.** An unknown segment expands to `FALSE` (well-formed in a
  WHERE slot) and is surfaced as `Unknown SEGMENT(x)` before the cell runs.
- **Managed in the Semantic panel.** Add / list / delete + a "used by N cells"
  count, alongside measures + dimensions. The **View as code** JSON now
  round-trips `{measures, dimensions, segments}`.
- **Pure macro expansion.** Emits a cell you run (Hard NOT #4); never auto-runs;
  the definition lives in the workbook description, never the data.

## Compatibility

- **New optional `segments` field** on `.naklidata` (mirrors how `dimensions`
  shipped in v1.4). Pre-M2 files round-trip cleanly; **no breaking change**, no
  format-version bump.
- No new dependencies.
